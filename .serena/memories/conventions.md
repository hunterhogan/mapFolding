- Preserve existing formatting and minimal diffs. `.editorconfig`: Python/source files use tabs; `.md/.toml/.yml` use 2 spaces; LF endings; final newline; trailing whitespace trimmed.
- Avoid autoformatters unless the repo already standardizes them. Preserve manual alignment, quote style, slicing style, and existing whitespace when correctness does not require change.
- Python identifiers: camelCase full words; no abbreviations or single-character bindings; callables use explicit subject-verb-object naming when practical; preserve domain casing and serialized/key names.
- Do not alias `numpy` as `np` or `pandas` as `pd`.
- Prefer domain-driven, functional/post-defensive code: pure transforms when possible, trust validated invariants internally, fail fast on violated assumptions, avoid silent recovery and redundant guards in hot paths.
- Prefer Python-native solutions over subprocesses, external config formats, env vars, or CLI frameworks unless an external boundary truly requires them. Do not introduce `shutil.rmtree` casually.
- Types: precise everywhere, PEP 585/604 style, no `Any`/`object`, favor local domain types and `collections.abc` abstractions for read-only inputs.
- Diagnostics: assign `message: str` first, then raise/warn/log; wording should be first-person observed state with runtime values.
- Comments/docstrings: do not add unless requested or truly needed. If docstrings are required, use the modified NumPy-style rules from `.github/HH-docstrings.md`; comments should explain non-obvious why, not restate code.
- Syntactic clarity: use semantic identifiers for ambiguous numeric adjustments (`inclusive`, etc.), prefer `<`/`<=` orientation, import from public APIs, and prefer absolute imports over relative imports inside the package.
- For pytest structure and fixture placement, read `mem:testing`. For `_e` indexing/bitset invariants, read `mem:_e/core`.
