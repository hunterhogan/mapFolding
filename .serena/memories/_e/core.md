- `mapFolding/_e/` is an internal experimental elimination-algorithm workbench, not the stable package API. Public-ish re-exports live in `mapFolding/_e/__init__.py`; stable package APIs still live outside `_e`.
- Front door: `mapFolding/_e/basecamp.py`. `eliminateFolds(...)` dispatches by `flow` (`None`/`elimination`, `constraintPropagation`, `crease`) and handles persistence plus CPU limits.
- Core state: `mapFolding/_e/dataBaskets.py::EliminationState`. It centralizes derived values from `mapShape`; `EliminationState.foldsTotal` equals `groupsOfFolds * Theorem2aMultiplier * Theorem2Multiplier * Theorem3Multiplier * Theorem4Multiplier`, and `Theorem2aMultiplier` is initialized to `leavesTotal`.
- Algorithms live in `mapFolding/_e/algorithms/` and expose `doTheNeedful(state, workersMaximum) -> EliminationState`.
- `elimination.py` is the baseline theorem/pinning flow; `constraintPropagation.py` is OR-Tools CP-SAT with optional research CSV dumps; `eliminationCrease.py` is specialized to `(2,) * n`; `iff.py` holds feasibility/forbidden-inequality checks.
- Supporting modules are a semiotic contract: `_beDRY.py`, `_leafDomains.py`, `_leafOptionsAtPile.py`, `_measure.py`, `_semiotics.py`, `_theTypes.py`, `filters.py`, `pinIt.py`, `pin2上nDimensions*.py`. Prefer imports from `mapFolding._e` when the symbol is re-exported.
- `_e` leaf/pile indexing is 0-based. Do not normalize to the main package's 1-based conventions while working inside `_e`.
- Pile ranges use `gmpy2.mpz` bitsets with a sentinel bit. Use helpers like `makeLeafOptions`, `getIteratorOfLeaves`, `JeanValjean`, and `bifurcatePermutationSpace` instead of re-encoding the representation.
- Keep algorithm selection centralized in `_e/basecamp.py`. Avoid extra guards/branches in hot paths.
- Read `mem:_e/pattern_discovery` when deriving generalized formulas from empirical/static tuples.
